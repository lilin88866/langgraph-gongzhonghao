# 来源整理

## 使用场景

在文章发布前、正文生成后，或用户提供链接、笔记、转写稿、截图、报告、摘录内容并需要整理为公众号来源说明时使用。

## 输入信息

- `topic`：文章选题。
- `article_html`：当前正文。
- `sources`：来源素材、链接、摘录、摘要、截图或笔记。
- `image_urls`：已使用或计划使用的图片。
- `manual_notes`：用户关于署名、版权或发布的补充要求。

如果缺少来源素材，必须明确标记这篇草稿需要补充来源。

## 整理规则

- 区分事实来源、灵感参考、图片来源和用户笔记。
- 不要编造来源名称、链接、发布时间、作者名或授权方式。
- 对缺少支撑的正文声明标记“需要人工复核”。
- 对搬运段落、截图、图表、产品 logo、图片等标记版权复核。
- 来源图片如果是流程图、架构图、表格截图、对比图或系统框图，默认不建议直接使用，应标记为“参考其信息结构，重新绘制”。
- 图片复核要区分两件事：`事实信息能否参考` 和 `图片本身能否发布`。前者可作为灵感，后者通常需要授权或重画。
- 对结构图要提取信息结构，例如对象、层级、箭头、对比维度、输入输出，再给出重画方向。
- 如果只有图片 URL、文件名或 alt/title，不能声称已经看见图片细节；需要标记“需人工补充截图 OCR 或图片内容描述”。
- 优先输出编辑可直接使用的简洁来源说明。
- 除非用户要求，不要写成学术论文引用格式，要服务于发布流程。

## 输出格式

必须使用下面这个结构：

```markdown
### 来源完整度

高 / 中 / 低

### 来源清单

1. 名称：
   类型：事实来源 / 灵感参考 / 图片来源 / 用户笔记 / 待补充
   链接：
   用途：
   可信度：
   需要复核：

### 正文声明复核

1. 声明：
   位置：
   当前依据：
   复核建议：

### 图片与版权复核

1. 图片/素材：
   当前来源：
   风险：
   建议处理：
   信息结构提取：
   重画方向：
   需要人工补充：

### 可放入文末的来源说明

[一段适合放在公众号文末的来源说明。如果来源不足，要说明暂不适合作为最终来源说明发布。]

### 发布前来源清单

- [ ] 关键事实已有来源。
- [ ] 图片、截图、图表确认可使用。
- [ ] 引用内容没有大段搬运。
- [ ] 产品名、时间、版本、数据已复核。
- [ ] 文末来源说明已人工确认。
```

## 质量检查

输出前检查：

- 用户提供的每个来源都出现在来源清单里。
- 缺失来源已经明确指出。
- 版权和图片风险清楚可见。
- 来源说明没有过度声称确定性。
