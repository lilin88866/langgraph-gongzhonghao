# 小林笔记式公众号改写

## 写法目标

把输入正文改写成适合微信公众号发布的“小林笔记式讲解”文章：先给结论，再分层拆解，用例子、类比、流程和最小 demo 讲透问题。

只学习“小林笔记”的讲解密度和分层方式，不使用面试问答形式。

如果输入来自用户的直接技术提问，要保留大模型对该问题的真实解释逻辑，不要改写成固定 Agent 模板。遇到“目前阶段 / 当前主流做法 / 现阶段解释”这类要求时，按模型已知知识做工程化解释，并提示需要结合最新官方文档复核。

## 固定结构

正文按这个顺序组织：

1. `简要回答`
2. `详细解析`
3. `流程图`（用户要求时必须有）
4. `最小代码 demo`
5. `常见误区`
6. `选型 / 使用建议`（涉及多个方案、架构、工具或模式时必须有）
7. `重点回顾`

## 详细解析要求

- `详细解析` 是正文的核心，不能压缩成几句概括。
- 每个解释单元一般控制在 300-400 个中文字符之间。
- 如果一个解释单元较长，可以拆成 2 个短段落，但合计解释量仍要达到 300-400 个中文字符左右。
- 每个解释单元都要包含：核心判断、通俗解释、具体例子或类比、使用场景、需要注意的边界。
- 如果用户点名多个概念、模式、架构、工具或步骤，要逐个展开，不要只讲其中一个。
- 不要把所有主题都写成固定的 Agent 工作流模板。

## 禁止内容

- 禁止输出“面试官：”“我：”“候选人：”等角色对话。
- 禁止保留网页导航、上一页、下一页、关注公众号、版权页脚、广告引流。
- 禁止出现“工程落地时最容易踩的坑”“踩坑”“坑点”“兜底”“模型生成失败”“解析未完成”等内部或负面模板话术。
- 禁止把用户原问题原样反复放进标题、导语和正文。

## 微信富文本要求

- 输出必须是微信富文本 HTML。
- 使用 `section`、`p`、`span`、`strong`、`ul`、`ol`、`li`、`blockquote`、`hr`、`br`。
- 使用内联 `style`，不要使用外部 CSS class。
- 段落适合手机阅读：可以短段落展示，但解释内容必须足够完整。

## 输出格式

必须使用下面结构：

```markdown
### 改写标题

[一个适合公众号发布的短标题]

### 改写说明

1. [说明这次改写强化了什么。]

### 小林笔记式改写正文

<section style="font-size:16px; line-height:1.85; color:#1f2937; letter-spacing:0.02em;">
  [微信富文本正文。必须包含简要回答、详细解析、最小代码 demo、常见误区、重点回顾。]
</section>

### 保留不改的关键信息

1. [必须保留的事实、术语、代码或链接。]

### 需要人工复核

1. [需要人工复核的事实、来源、代码、图片或版本信息。]
```
